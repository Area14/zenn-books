## Gemma 4の位置づけ

Gemma 4は、Google DeepMindがGemini 3系の研究成果をベースに構築した**オープンウェイトモデル**です。ライセンスは Apache 2.0 と完全な商用フリー。オープンソースLLMの中では、モデルサイズ・性能・ライセンスのバランスが非常に取れた立ち位置に収まっています。

特筆すべきは、単にパラメータ数を増やした後継ではなく、**モデル体制そのものを再設計している**点です。

## 4モデル体制：用途別ラインナップ

| モデル | パラメータ | 推奨VRAM | 想定用途 |
|---|---|---|---|
| Gemma 4 E2B | 2B | 4GB〜 | モバイル／組み込み／スマホ級端末 |
| Gemma 4 E4B | 4B | 6GB〜 | ノートPC／ミドルレンジGPU |
| Gemma 4 26B | 26B (MoE) | 16GB〜 | デスクトップGPU／主力運用 |
| Gemma 4 31B | 31B (Dense) | 24GB〜 | ワークステーション／本格運用 |

**E系（Edge）モデル**は端末側推論を明確に意識した軽量設計、**26B** は Mixture-of-Experts でコスト効率、**31B** は Dense で精度最大化という役割分担が明確です。この「選べる設計」が Gemma 4 の最大の特徴と言ってよいと思います。

## 前世代からの進化ポイント

### 1. Function Calling が使い物になった

もっとも大きい進化が Function Calling の精度向上です。Gemma 3 系ではベンチマークで **6.6%** だった関数選択・引数構築の正答率が、Gemma 4 では **86.4%** にまで跳ね上がりました。これにより、ローカルモデルでもツール呼び出し型のエージェントが実運用ラインに乗ります。

### 2. 256K トークンのコンテキスト長

長文処理が本格的に可能になりました。256Kトークンは、英語換算でおよそ19万語、日本語換算で文庫本数冊分の長さです。RAGだけに頼らず「読ませる」運用も現実的になります。

### 3. マルチモーダル入力

画像・音声の入力をネイティブでサポートします。PDFに混ざった図表やスキャン文書、会議音声といった、従来は OCR + 別モデル処理に分割していたワークフローを、1モデルで完結させられます。

### 4. 140言語対応

主要言語に対する処理品質が前世代より底上げされています。日本語も、実用上は英語と遜色ない応答が返ります。

## どこで動くのか

- **Ollama**: 最も手軽。CLIも設定ファイルも最小限
- **llama.cpp**: 量子化の自由度が高い。エッジ・CPU運用向け
- **LM Studio**: GUIで完結。非エンジニアにも渡せる

E2B なら一般的なノートPCでも動き、31B でも24GB VRAMの単一GPUで実用速度が出ます。

## どの仕事をローカルに移すべきか

Gemma 4 のスペックを前提に、私の現時点の判断軸は以下です。

| 仕事 | 移すべきか | 理由 |
|---|---|---|
| 社内ドキュメントに対するRAG検索 | ◎ 移す | データ外部送信回避・定常コスト削減 |
| 大量コード補完 | ◎ 移す | トークン量が多くAPI課金とリニアに効く |
| 画像・PDF一括解析 | ◎ 移す | マルチモーダル完結・バッチ処理 |
| フロンティアな推論タスク | △ 残す | 最上位クラウドモデルが依然優位 |
| 長期記憶・検索を伴うエージェント | ○ 併用 | ローカル Function Calling＋クラウド長期記憶 |

クラウド側の業務エージェント（例: Claude Managed Agent）と役割分担するのが、最も合理的に思えます。全部ローカルに倒すのではなく、**「ローカルに軸足を戻す」**というニュアンスです。
