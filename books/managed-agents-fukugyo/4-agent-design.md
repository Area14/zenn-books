前章では、Anthropic ConsoleでのAPIキー取得、CLIツール「ant」のインストール、そしてはじめてのエージェント作成までを完了しました。エージェントが動く「環境」は整いました。しかし、第3章で作ったエージェントは「親切に答えてね」程度のシンプルな指示しか持っていませんでしたよね。

この章では、エージェントの「頭脳」を設計する技術を身につける。システムプロンプトの書き方、ツールの選び方、実行環境（Environment）の設定、デバッグの方法——この4つを押さえれば、「なんとなく動く」エージェントから「狙い通りに動く」エージェントへと一段レベルが上がる。第5章以降の実践編でジャンル別のエージェントを構築するとき、ここで身につけた設計スキルがすべての土台になる。

## 4-1. よいシステムプロンプトの書き方

### 役割定義・制約条件・出力形式の3点セット

システムプロンプトとは、エージェントに「あなたは何者で、何をすべきか」を伝える指示書です。第3章のハンズオン3では「あなたは自己紹介の達人です」という短い指示だけでしたが、副業で実用レベルのエージェントを作るには、もっと構造化された指示書が必要です。

よいシステムプロンプトは、**3つの要素**で構成されます。

**1. 役割定義（Role）：「あなたは誰か」**

エージェントにペルソナ（役割）を与えます。「あなたはSEOに詳しいWebライターです」「あなたは市場調査のプロフェッショナルです」のように、具体的な専門性を指定するのがポイントです。「なんでもできる便利なアシスタント」よりも、専門分野を絞ったほうが出力の品質が上がります。

たとえば、ブログ記事を書くエージェントなら：

```
あなたはSEOに精通した日本語ブログライターです。
読者は副業に興味がある30〜40代の会社員です。
専門用語を使う場合は、必ず平易な言い換えを添えてください。
```

**2. 制約条件（Constraints）：「やってはいけないこと・守るべきルール」**

エージェントの行動範囲を明確に区切ります。制約がないと、エージェントは良かれと思って余計なことをする可能性があります。

```
## 制約条件
- 文字数は2,000〜3,000字の範囲に収めること
- 医療・法律に関するアドバイスは行わないこと
- 競合他社の誹謗中傷は含めないこと
- 情報の出典が不明な場合は「※要確認」と注記すること
```

**3. 出力形式（Output Format）：「どんな形で結果を出すか」**

エージェントの出力フォーマットを具体的に定義します。「いい感じに書いて」では、毎回違う形式で出力されてしまいます。

```
## 出力形式
以下のMarkdown構造で記事を出力してください：
- # タイトル（32文字以内、キーワードを含む）
- ## リード文（3文以内で記事の要点をまとめる）
- ## 見出し1〜3（各セクション400〜600字）
- ## まとめ（3つの箇条書き）
```

この3点セットを守るだけで、エージェントの出力は劇的に安定します。「なんか微妙な結果が返ってくるな」と感じたとき、原因の8割はシステムプロンプトの設計不足です。

### ChatGPTプロンプトとの違い：一度きりではなく繰り返し使う前提

「それって、ChatGPTのプロンプトと同じじゃないの？」——3点セットの考え方は確かにChatGPTでも使えるテクニックだ。しかし、エージェントのシステムプロンプトには決定的な違いがある。

**ChatGPTプロンプト**は「1回きりの指示」です。うまくいかなければ、その場で「もうちょっとカジュアルに」と追加指示を打てます。

**エージェントのシステムプロンプト**は「何十回、何百回と繰り返し使う前提」で書きます。一度設定すれば、Sessionを作るたびに同じプロンプトで自律的に動きます。途中で「ちょっと違うな」と軌道修正する余地が少ないのです。

つまり、**あいまいさを残してはいけない**。これが最大の違いです。

具体的なチェックポイントを挙げます。

- **「適切に」「いい感じに」は禁止**。「2,000字以上3,000字以下」「見出しは3つ以上」のように数値で指定する
- **例外処理を明記する**。「キーワードの検索結果が5件未満の場合は、関連キーワードに変更して再検索すること」
- **優先順位をつける**。「文字数制限と品質が矛盾する場合は、品質を優先すること」

ChatGPTでは「曖昧でもなんとかなる」ことが多いですが、エージェントでは「曖昧さ＝不安定さ」です。最初に少し手間をかけてプロンプトを設計することで、あとの運用がずっと楽になります。1回きりの質問なら「よしなにやって」で済みますが、エージェントのシステムプロンプトは「業務マニュアル」として書く——これがコツです。

## 4-2. ツール選択の考え方 — 最小限から始める

### 副業タスク別おすすめツール構成

第2章で紹介した8つの標準ツール（`agent_toolset_20260401`）を覚えていますか？ Bash、Read、Write、Edit、Glob、Grep、Web search、Web fetchの8種類です。「全部使えるようにしておけばいいんでしょ？」と思うかもしれないが、それは逆効果だ。

ツールを増やすと、エージェントの「選択肢」が増えます。選択肢が増えるということは、判断に迷うケースも増えるということです。結果として、余計なツール呼び出しが発生し、セッション時間が延びてコストが増え（API利用料がトークン量に応じて加算されます）、意図しない動作が起きるリスクも高まります。

鉄則は「**必要最小限のツールだけを使わせる**」です。

副業タスク別に、おすすめのツール構成を整理しました。

| 副業タスク | 必須ツール | あると便利 | 不要 |
|-----------|-----------|-----------|------|
| **ブログ記事作成** | Web search, Write | Web fetch, Edit | Bash, Glob, Grep |
| **市場調査レポート** | Web search, Web fetch, Write | Bash（集計用） | Edit, Glob, Grep |
| **SNS投稿作成** | Web search, Write | — | Bash, Read, Edit, Glob, Grep, Web fetch |
| **データ整理・分析** | Read, Write, Bash | Glob, Grep | Web search, Web fetch |
| **メール返信テンプレ** | Read, Write | — | Bash, Glob, Grep, Web search, Web fetch |

たとえば、ブログ記事を書くエージェントに「Bash」を与えると、エージェントが「Pythonで文字数をカウントしてみよう」などと余計なコマンド実行をすることがあります。記事を書くだけならWeb search（リサーチ用）とWrite（記事保存用）の2つで十分です。

### 「Web search + Write」だけで始める最小構成

初心者の方には、まず**Web search + Write**の2ツール構成から始めることを強くおすすめします。この最小構成だけで、以下のことが実現できます。

- キーワードでWeb検索して競合記事や最新情報を収集
- 収集した情報をもとに文章を生成
- 結果をMarkdownファイルとして保存

ant CLIでエージェントを作成するとき、ツールは`tools`パラメータで指定します。現在のManaged Agentsでは`agent_toolset_20260401`を指定すると8つのツールがまとめて有効になります。

```bash
# 標準ツールセットを有効にしてエージェントを作成
ant agents create \
  --name "blog-writer" \
  --model "claude-sonnet-4-6" \
  --system-prompt "あなたはSEOブログライターです..." \
  --tools '{"type": "agent_toolset_20260401"}'
```

このコマンドでは8つの標準ツールがすべて有効になりますが、最小構成で運用したい場合はシステムプロンプトの制約条件で「Web searchとWriteのみを使用してください。他のツールは使わないでください」と明記すれば、事実上の最小構成として運用できます。エージェントはシステムプロンプトの指示に従うので、使わせたくないツールは「禁止」と書けばOKです。

**ツール追加の順序のおすすめ：**

1. **Web search + Write**（記事作成の基本）
2. **+ Web fetch**（特定URLの内容を取得。競合記事の詳細分析に便利）
3. **+ Read**（既存ファイルの読み込み。テンプレートや過去データの参照に）
4. **+ Edit**（既存ファイルの部分修正。リライト作業の自動化に）
5. **+ Bash**（コマンド実行。データ集計やスクリプト実行が必要な場合のみ）

この順序で段階的にツール許可を広げれば、各ツールの効果を実感しながら安全にエージェントの能力を拡張できます。

より高度なカスタムツール連携には、MCP（Model Context Protocol、第2章参照）の活用も視野に入る。第9章のマルチエージェント構成に慣れてから取り組むのがおすすめだ。

## 4-3. Environmentの設定 — ネットワークとパッケージ

### ネットワーク制限あり/なしの使い分け

第2章で学んだEnvironment（実行環境）は、エージェントが動く「作業場」だ。第3章ではConsole画面で「Internet Access: ON」に設定した。ここでは、もう少し踏み込んだネットワーク設定を整理する。

Managed AgentsのEnvironmentには、ネットワーク設定が2つのモードで用意されています。

**1. unrestricted（制限なし）**

エージェントがインターネット上のあらゆるサイトにアクセスできるモードです。Web searchやWeb fetchを使うエージェントは、基本的にこのモードを選びます。

```bash
# ネットワーク制限なしのEnvironmentを作成
ant environments create \
  --name "web-enabled" \
  --config '{"type": "cloud", "networking": {"type": "unrestricted"}}'
```

**2. limited（制限付き）**

パッケージマネージャー（pip、npmなど）への接続と、個別に許可したドメインのみアクセスできるモードです。エージェントが意図しないサイトにアクセスすることを防ぎたい場合や、クライアントの機密データを扱う場合に使います。

```bash
# 特定のドメインのみ許可する設定
ant environments create \
  --name "restricted-env" \
  --config '{
    "type": "cloud",
    "networking": {
      "type": "limited",
      "allowed_hosts": ["api.example.com", "data.example.com"]
    }
  }'
```

**副業での使い分けの目安：**

| 用途 | おすすめ設定 | 理由 |
|------|------------|------|
| ブログ記事作成 | unrestricted | 多様なサイトからの情報収集が必要 |
| 市場調査 | unrestricted | 多様なサイトへのアクセスが必要 |
| 社内データ分析 | limited | 外部への情報流出を防止 |
| テンプレート加工 | limited | ネット不要、安全性優先 |

副業で使う大半のケースでは「unrestricted」を選べば問題ありません。ただし、クライアントの顧客リストや売上データなど機密情報を扱う場合は「limited」で明示的にアクセス先を制限しましょう。セキュリティへの配慮は、クライアントからの信頼にもつながります。

> **注意点**: 制限付きネットワーク（`limited`）を選んだ場合、`allowed_hosts`に必要なドメインが含まれていないと、ツールが**サイレント・フェイル**（エラーメッセージを出さずに失敗）することがあります。「Web fetchがなぜか動かない」というときは、まずこの設定を確認してください。

### 必要なパッケージの指定方法

Environmentでは、エージェントが使うパッケージ（ソフトウェアの部品）を事前にインストールしておけます。たとえば、データ分析にPythonのpandasライブラリを使いたい場合、セッション中に毎回インストールする必要はありません。Environment作成時に`packages`フィールドで指定するだけです。

```bash
# PythonとNode.jsのパッケージを事前インストール
ant environments create \
  --name "data-analysis-env" \
  --config '{
    "type": "cloud",
    "packages": {
      "pip": ["pandas", "numpy", "matplotlib"],
      "npm": ["csv-parser"]
    },
    "networking": {"type": "unrestricted"}
  }'
```

`packages`フィールドでは、パッケージマネージャーごとにインストールしたいパッケージ名をリストで指定します。`pip`（Python用）と`npm`（Node.js用）が使えます。指定したパッケージは同じEnvironmentを使うセッション間でキャッシュされるので、2回目以降の起動が高速です。

**副業タスク別のおすすめパッケージ：**

- **データ分析**: `pip: ["pandas", "numpy", "matplotlib"]`
- **PDF生成**: `pip: ["reportlab", "weasyprint"]`
- **Webページの解析**: `pip: ["beautifulsoup4", "requests"]`

ただし、副業の多くの用途——ブログ記事作成、SNS投稿生成、リサーチなど——では、追加パッケージなしのデフォルト環境で十分です。クラウドコンテナにはPython、Node.jsなどの主要ランタイムが最初から入っています。パッケージが必要になるのは、データ分析や帳票生成など、Pythonスクリプトを実行する場面に限られます。「必要になってから追加する」のスタンスで問題ありません。

## 4-4. エージェントの動作を確認・デバッグする方法

### Eventsの読み方

エージェントを作って動かしてみたけれど、「思った結果と違う」「途中でおかしくなった」——こんな場面は必ず出てきます。そんなとき頼りになるのが**Events（イベントログ）**です。

第2章で学んだように、Eventsはエージェントとのやり取りの全記録です。あなたの指示、エージェントの思考過程、ツールの呼び出し結果——すべてが時系列で記録されています。Console画面でセッションをクリックすると確認できますし、CLIでも取得できます。

```bash
# セッションのイベント一覧を取得
ant sessions events --session-id session_xxxxxxxx
```

イベントには主に3つの種類があります。

**1. user（ユーザーからの指示）**

あなたがエージェントに送ったメッセージです。「キーワード『副業 AI 始め方』でブログ記事を書いて」のような指示内容が記録されます。

**2. assistant（エージェントの応答・行動）**

エージェントが何をしたかの記録です。テキスト応答だけでなく、「Web searchツールを呼び出した」「Writeツールでファイルを保存した」などのツール実行の詳細も含まれます。ここを読むと、エージェントが「何を考えて、何をしたか」が手に取るようにわかります。

**3. tool_result（ツールの実行結果）**

ツールが返した結果です。Web searchの検索結果一覧や、Writeツールの成功/失敗メッセージなどが記録されます。エラーが起きた場合は、ここにエラーメッセージが残っています。

デバッグの基本は、**イベントを上から順に読むこと**です。Webリサーチエージェントを例にとると、イベントは次のような流れで記録されます。

```
1. [user]         「AIエージェントの市場規模を調べてください」
2. [assistant]     → Web searchツールを呼び出し
3. [tool_result]   検索結果10件を取得
4. [assistant]     → Web fetchツールで詳細ページを取得
5. [tool_result]   ページ内容を取得
6. [assistant]     → Writeツールでレポートを保存
7. [tool_result]   ファイル書き込み完了
8. [assistant]     「調査結果をresearch_memo.mdに保存しました」
```

「どの時点で意図と違う方向に進んだか」を特定できれば、修正すべきポイントが見えてきます。

### うまく動かないときのチェックポイント

エージェントが期待通りに動かないとき、闇雲に設定を変えるのは逆効果です。以下の5つのチェックポイントを上から順に確認してください。

**チェック1：システムプロンプトは明確か？**

最も多い原因です。「適切に」「わかりやすく」などの曖昧な表現がないか確認しましょう。「3つの箇条書き、各50字以内で」のように数値・条件・形式を具体的に指定しているかチェックしてください。

**チェック2：ツールは正しく設定されているか？**

Web searchを使いたいのにツールが有効になっていない、というミスは意外と多いです。Console画面でAgent定義のツール設定を確認しましょう。

**チェック3：Environmentのネットワーク設定は合っているか？**

制限付きネットワーク（`limited`）を選んでいる場合、`allowed_hosts`にアクセス先のドメインが含まれているか確認してください。前述の「サイレント・フェイル」が起きていると、エラーも出ずにただ失敗します。

**チェック4：出力形式の指定は明確か？**

「レポートを書いて」だけでは、エージェントが毎回違う形式で出力します。Markdown形式、箇条書き、見出しの構造まで具体的に指定しましょう。

**チェック5：タスクが複雑すぎないか？**

1つのエージェントに「調べて、まとめて、翻訳して、校正して」と詰め込むと、途中で品質が落ちることがあります。タスクを分割して段階的に処理させることを検討してください（これは第9章のマルチエージェント構成で詳しく解説します）。

> **ワンポイントアドバイス**: デバッグ作業そのものにもコスト（セッション時間＋API利用料）がかかります。エージェントに長いタスクを実行させる前に、まず短い指示で「意図通りに動くか」を確認するテスト実行の習慣をつけましょう。「記事を10本書いて」の前に「記事を1本書いて」で出力を確認する。これだけでデバッグコストを大幅に抑えられます。



### ハンズオン5：「Webリサーチエージェント」を設計・作成する（テーマ自由）

**ゴール**: この章で学んだ設計テクニック（3点セット・ツール選択・Environment設定）を使って、Webリサーチからレポート出力までを行うエージェントを自分の力で設計・作成する

**準備**: Anthropic Consoleにログイン済み、antインストール済み、APIキー設定済み（第3章のハンズオン3・4が完了していること）

1. まず、リサーチしたいテーマを1つ決めてください。あなたの副業に関連するテーマがおすすめです。例：「2026年のAI副業トレンド」「在宅ワークに最適なツール比較」「初心者向け投資信託の選び方」

2. この章で紹介した「3点セット」に従い、システムプロンプトを設計します。以下のテンプレートを参考に、自分のテーマに合わせてカスタマイズしてください。

```
あなたはWebリサーチの専門家です。

## 役割
指定されたテーマについてWeb検索を行い、最新の情報を収集して
構造化されたリサーチレポートを作成してください。

## 制約条件
- 情報源は3つ以上のサイトから収集すること
- 1年以内の情報を優先すること
- 事実と意見を明確に区別すること
- 出典が不明な情報には「※要確認」と注記すること
- 日本語で出力すること

## 出力形式
以下のMarkdown形式でレポートをファイル出力してください：
# リサーチレポート：{テーマ}
## 調査日：{今日の日付}
## 要約（3行以内）
## 主な発見（箇条書き5〜10個）
## 詳細分析（各発見について2〜3文で解説）
## 情報源リスト（URLを明記）
```

3. Console画面（またはant CLI）でエージェントを作成します。CLIの場合：

```bash
ant agents create \
  --name "web-researcher" \
  --model "claude-sonnet-4-6" \
  --system-prompt "（上で設計したプロンプトを貼り付け）" \
  --tools '{"type": "agent_toolset_20260401"}'
```

4. Environmentを作成します（Web検索を使うのでネットワークはunrestricted）：

```bash
ant environments create \
  --name "research-env" \
  --config '{"type": "cloud", "networking": {"type": "unrestricted"}}'
```

5. Sessionを作成してエージェントを起動します：

```bash
ant sessions create \
  --agent-id agent_xxxxxxxx \
  --environment-id env_xxxxxxxx \
  --message "「あなたが選んだテーマ」について調査してください"
```

6. Console画面の「Sessions」でセッションを開き、Eventsタブでエージェントの動作を確認してください。どのツールをどの順番で使ったかを見てみましょう。

**確認ポイント**: 以下の3つを満たしていれば成功です。
- レポートが指定したMarkdown形式で出力されている
- 3つ以上の情報源からの情報が含まれている
- 要約・主な発見・詳細分析・情報源リストの全セクションが揃っている

> **つまずきポイント**: レポートの品質が「浅い」と感じたら、制約条件に「各トピックについて具体的なデータや数字を含めること」を追加してみてください。出力形式が崩れる場合は、テンプレート部分をより詳細に書くか、「例」を1つ含めると改善されます。Web searchの結果が英語ばかりの場合は「日本語のソースを優先して検索してください」とプロンプトに追加しましょう。

**エージェント設計のフローチャート**


エージェントを作る前に、以下の流れで設計を進めましょう。「いきなりプロンプトを書く」のではなく、「何をさせるか」から逆算するのがコツです。

```
 ① 自動化したいタスクを決める
 │  「ブログ記事を書く」「市場調査する」など
 │
 ▼
 ② 入力と出力を定義する
 │  入力：キーワード3つ → 出力：Markdown記事1本
 │
 ▼
 ③ 必要なツールを選ぶ（最小限）
 │  Web search + Write で十分？
 │  Web fetchやBashも必要？
 │
 ▼
 ④ システムプロンプトを書く（3点セット）
 │  役割定義 → 制約条件 → 出力形式
 │
 ▼
 ⑤ Environmentを設定する
 │  ネットワーク：unrestricted or 制限付き？
 │  パッケージ：追加が必要？
 │
 ▼
 ⑥ テスト実行（短いタスクで）
 │  いきなり本番ではなく、小さなタスクで確認
 │
 ▼
 ⑦ Eventsを確認してプロンプトを改善
 │  意図通りか？→ Yes → 本番運用へ
 │             → No  → ④に戻って修正
 │
 ▼
 ⑧ 本番運用開始
    成功したプロンプト＝資産。何度でも使い回せる！
```

**読み方のポイント**: このフローで最も時間をかけるべきは④（システムプロンプト）と⑥→⑦（テスト＆改善）のループです。最初から完璧なプロンプトが書ける人はいません。テスト→Eventsで確認→修正を2〜3回繰り返すことで、実用レベルのエージェントが完成します。この改善ループにかかるコストは、1回あたりせいぜい$0.50（約75円）程度。コーヒー1杯より安い投資で、何度でも使える「自動化の仕組み」が手に入ると考えれば、安いものです。

:::


【チェックリスト】エージェント作成前の10項目チェック

エージェントを作成する前に、以下の10項目を確認してください。すべてにチェックが入れば、安心して作成に進めます。

- [ ] **1. 目的が明確か？** — 「何を自動化するか」を1文で説明できる
- [ ] **2. 入力が定義されているか？** — エージェントに渡す情報（キーワード、データ等）を決めた
- [ ] **3. 出力が定義されているか？** — 成果物のファイル形式・構造・文字数を決めた
- [ ] **4. 役割定義は具体的か？** — 「便利なアシスタント」ではなく専門的なペルソナを設定した
- [ ] **5. 制約条件を3つ以上書いたか？** — やってはいけないこと・守るべきルールを明記した
- [ ] **6. 出力形式を具体的に指定したか？** — 「いい感じに」ではなく、テンプレートで構造を明示した
- [ ] **7. ツールは最小限か？** — 本当に必要なツールだけを選んだ（まずはWeb search + Writeから）
- [ ] **8. ネットワーク設定は適切か？** — Web検索が必要ならunrestricted、機密データなら制限付き
- [ ] **9. テスト用の短い指示を用意したか？** — 本番タスクの前に、小さなタスクで動作確認する準備ができている
- [ ] **10. コスト感覚を持っているか？** — 1回のテスト実行は$0.05〜0.30程度。本番タスクでもランタイム$0.08/時間＋API利用料が目安（コストの大部分はAPI利用料）

> **ワンポイント**: 特に重要なのは1・3・6の3項目です。「目的」「出力」「出力形式」——この3つが曖昧なまま作り始めると、何度やり直しても満足のいく結果になりません。逆に、この3つが明確であれば、システムプロンプトはほぼ自動的に書けます。



## まとめ

- **システムプロンプトは「役割定義・制約条件・出力形式」の3点セットで構成する**。曖昧な表現は避け、数値・条件・フォーマットを具体的に指定する。ChatGPTと違い「繰り返し使う」前提で設計するのがエージェントのプロンプトの本質
- **ツールは最小限から始める**。副業のほとんどのタスクは「Web search + Write」の2ツールで十分にスタートできる。ツールが増えるとエージェントが迷い、コストも上がる
- **Environmentはネットワーク設定がカギ**。Web検索を使うならunrestricted、機密データを扱うならlimitedで制限する。packagesフィールドで必要なパッケージを事前インストールすれば起動も高速化できる
- **デバッグはEventsを読むことから始める**。user→assistant→tool_resultの流れを追えば、エージェントの全行動が可視化される。「どこで意図とズレたか」を特定し、システムプロンプトを改善するサイクルを回す
- **設計シートを埋めてから作り始める**。「いきなりプロンプトを書く」のではなく、目的→入出力→ツール→プロンプトの順で設計する。テスト→改善を2〜3回繰り返せば実用レベルのエージェントが完成する

次の第5章では、いよいよ副業ジャンル別の実践に入ります。最初に取り上げるのは「ライティング副業の自動化」です。ブログ記事の量産、電子書籍の執筆補助、品質管理、そしてコスト試算まで——この章で身につけた設計スキルを使って、実際に稼げるエージェントを構築していきます。「設計の理論」から「お金を生む実践」へ。ここからが本番です。
