第4章では、エージェント設計の基本——システムプロンプトの「3点セット」（役割定義・制約条件・出力形式）やツール選択の考え方——を学んだ。「設計図の描き方」は手に入った。

ここからは第3部、副業ジャンル別の実践編だ。最初に取り上げるのは「ライティング副業」。ブログ記事の執筆、電子書籍（Kindle）の出版——文章を書いて収益を得る副業は、AIエージェントとの相性が抜群だ。リサーチ→構成→執筆→校正という工程が明確で、自動化のパイプラインを組みやすいからである。

この章では、ブログ記事作成エージェントの構築から電子書籍の執筆補助、品質管理の仕組み、コスト試算まで、ライティング副業を自動化する実践的なノウハウを詰め込んだ。第4章で学んだ設計スキルを、ここで一気に「実戦投入」する。

## 5-1. ブログ記事作成エージェントを構築する

ブログ記事を書く作業を分解すると、大きく3つのステップに分かれます。キーワードリサーチ（何について書くかを決める）、構成案の作成（記事の骨組みを考える）、本文執筆（実際に文章を書く）です。手動でやれば1記事あたり2〜3時間。これをエージェント1つで自動化する方法を見ていこう。

### キーワードリサーチ → 構成案 → 本文執筆の自動化パイプライン

エージェントによるブログ記事作成の流れは、次の3ステップで構成します。

**ステップ1：キーワードリサーチ**

あなたがメインキーワード（例：「副業 在宅 おすすめ」）を渡すと、エージェントがWeb searchツールを使って以下を自動で調査します。

- 検索上位10記事のタイトルと見出し構成
- 関連キーワード（サジェストキーワード）の収集
- 競合記事の文字数・構成パターンの分析

**ステップ2：構成案の作成**

リサーチ結果をもとに、エージェントが記事の構成案（見出しの一覧）をMarkdownで作成します。SEO（検索エンジン最適化）に強い構成にするため、「検索上位の記事がカバーしている項目」を網羅しつつ、「競合が触れていない独自の切り口」を提案してくれます。

**ステップ3：本文執筆**

構成案に沿って、各見出しの本文を執筆します。指定した文字数、トーン、SEO要件を守りながら、一気にファイルとして保存します。

ここで重要なのは、この3ステップがすべて1つのセッション内で自動的に連続実行されるという点です。あなたがやるのは、最初にキーワードを渡すことだけ。あとはエージェントが判断しながら、リサーチ→構成→執筆を順番にこなし、完成した記事をファイルに書き出してくれます。

### システムプロンプト設計：トーン・文字数・SEO要件の指定

ブログ記事作成エージェントの品質を決めるのは、システムプロンプトの設計です。第4章で学んだ「3点セット」を、ブログ記事作成に特化させましょう。

**役割定義**では、「SEOに精通したプロのWebライター」と専門性を明確にします。漠然と「記事を書いて」ではなく、具体的な専門性を定義することで出力の質が大きく変わります。

**制約条件**では、以下の4つを指定します。

1. **文字数**: 「3,000〜5,000字」のように範囲で指定（「約4,000字」より守りやすい）
2. **トーン**: 「です・ます調」「フレンドリーな文体」など具体的に
3. **SEO要件**: 「メインキーワードをタイトル・h2見出し・本文冒頭100字以内に含める」
4. **禁止事項**: 「他サイトの文章をそのままコピーしない」「根拠のない数字を使わない」

**出力形式**では、「Markdown形式で、ファイル名は `{キーワード}.md`」のように保存形式まで指定します。そのままブログに入稿できる状態で受け取れます。

### Web searchツールでの競合記事リサーチ自動化

ブログ記事作成で最も時間がかかるのは、実はリサーチです。「このキーワードで検索上位に表示されている記事は何を書いているのか」「自分の記事に何を盛り込めば差別化できるのか」——これを手作業で調べると、1キーワードあたり30分〜1時間はかかります。

エージェントなら、Web searchツールで自動的に競合記事をリサーチできます。システムプロンプトに「メインキーワードで検索し、上位5記事の見出し構成を分析してください」と書くだけで、エージェントが検索を実行し、結果を整理してくれます。

さらに、Web fetchツールを組み合わせれば、特定URLの記事内容を取得して詳細な分析も可能です。

ただし注意点が1つ。Web searchには検索1,000回あたり$10の追加コストが発生します。1記事あたりの検索回数を3〜5回に制限するようシステムプロンプトで指定しておきましょう。月10記事なら検索30〜50回、コストは約45〜75円です。

## 5-2. 電子書籍（Kindle出版）の執筆補助エージェント

ブログ記事が「フロー型収入」（書き続けないと収益が止まる）だとすれば、電子書籍はまさに「ストック型収入」（一度出版すれば継続的に収益が発生する）です。Kindle出版の印税率は、KDP Select（Amazon独占配信）に登録すれば70%。500円の書籍が月100冊売れれば、月35,000円の不労所得になります。実際に、Kindle出版で月5〜10万円の副収入を得ている個人著者は少なくありません。

### 章ごとの執筆エージェント設計

電子書籍をまるごと1つのプロンプトで書かせようとする人がいますが、これは失敗のもとです。後半になるほど品質が落ち、内容の重複や論理の飛躍が発生します。

正しいアプローチは、**章ごとに別セッションで執筆する**ことです。

**1. アウトラインを先に作成する**

まず、書籍全体のアウトライン（目次＋各章の要点）をエージェントに作成させます。これは「企画エージェント」として、書籍テーマ・想定読者・ページ数を入力すれば自動で生成されます。

**2. 章ごとのセッションで執筆する**

アウトラインが確定したら、各章を個別のセッションで執筆します。このとき、システムプロンプトに以下の情報を含めることがポイントです。

- 書籍全体のアウトライン（全章の見出しと概要）
- 執筆済みの前章の末尾200〜300字（接続の自然さを保つため）
- 対象章の詳細な構成（小見出し・盛り込む要素）
- 書籍全体で統一すべきルール（用語、トーン、読者への呼びかけ方）

こうすることで、各章が独立しながらも一貫性のある書籍に仕上がります。

**3. 1章あたりの文字数を5,000〜8,000字に設定する**

電子書籍の1章は、5,000〜8,000字程度が読みやすいとされています。全10章の書籍なら50,000〜80,000字。エージェントの1セッションあたり15〜25分で1章が完成するので、全10章を書くのに3〜4時間分のセッション時間です。

### 一貫性チェック用の校正エージェント

章ごとに別セッションで書くと、どうしても表現のブレが生じます。「第3章では『ユーザー』と書いているのに、第5章では『利用者』になっている」「冒頭で『です・ます調』だったのに途中から『だ・である調』が混ざる」——こうした不一致を防ぐために、**校正エージェント**を別途用意します。

校正エージェントのシステムプロンプトには、以下の2つの機能を持たせます。

**機能1：用語統一チェック** — 「ユーザー」→OK、「利用者」→NGのように統一用語リストをシステムプロンプトに含め、全章で用語の揺れを検出させます。

**機能2：文体一貫性チェック** — 「です・ます調に統一」「1文80字以内」「箇条書きは体言止め」など文体ルールを定義し、逸脱箇所を指摘させます。

校正エージェントには「修正箇所のリスト」と「修正後の文章案」をセットで出力させましょう。あなたは確認して「OK」「NG」を判断するだけです。

## 5-3. 品質管理：AIっぽさを消す仕上げプロセス

エージェントに記事を書かせると、どうしても「AIっぽさ」が残ることがあります。「〜と言えるでしょう」「〜について解説します」「まず最初に」のような定型表現の多用、具体的なエピソードの欠如、やけに整理された見出し構成——慣れた読者には「これ、AIで書いたな」とすぐにバレます。

Googleの検索品質ガイドラインでも、2025年以降「E-E-A-T」（経験・専門性・権威性・信頼性）がますます重視されています。つまり、「実体験に基づいた独自の視点」がない記事は、たとえ正確な情報であっても検索上位に表示されにくくなっているのです。

では、どうすればAIっぽさを消せるのか。ここでは、エージェントと人間の「分業」による仕上げプロセスを設計します。

### 文体統一エージェントの設計

品質を上げるために、もう1つのエージェント——「文体統一エージェント（仕上げエージェント）」を用意します。このエージェントの役割は、執筆エージェントが書いた記事を「人間が書いたように」リライトすることです。

文体統一エージェントのシステムプロンプトには、以下のルールを含めます。

**削除する表現パターン:**
- 「〜と言えるでしょう」「〜ではないでしょうか」 → AI特有の推量表現
- 「まず最初に」「次に」「最後に」の機械的な連続使用 → 段落構成の単調さの原因
- 「〜について解説します」「〜を紹介します」 → メタ発言で読者が冷める
- 同じ文末の3回以上連続（「〜です。〜です。〜です。」） → 文末のバリエーション不足

**追加する要素:**
- 具体的な数字（「多くの人」→「約7割の人」）
- 短い体験談風のエピソード（「筆者も最初は半信半疑でしたが」）
- 読者への問いかけ（「心当たりはありませんか？」）

このエージェントに記事を通すだけで、AIっぽさはかなり軽減されます。ただし完全ではないので、最終的な「人間味」の付加は、次に説明する人間のチェックポイントで行います。

### 人間の最終チェックポイントの設定

エージェントによる自動化の最大の落とし穴は、「全部自動化しよう」と欲張ることです。実は、人間が5〜10分だけ手を加えるだけで、記事の品質は劇的に向上します。以下の3つのチェックポイントを設けましょう。

**チェック1：独自の知見を1つ追加する（3分）**

あなた自身の経験や意見を最低1か所追加します。「私が実際に試したところ〜」「個人的には〜がおすすめです」。たった1〜2文で、記事に「この人が書いた」という信頼性が生まれます。

**チェック2：事実情報のファクトチェック（3分）**

エージェントが生成した数字や固有名詞をざっと確認します。AIは「もっともらしいが間違っている」情報（ハルシネーション）を出すことがあります。特に料金・日付・統計データは要注意です。

**チェック3：冒頭と結論の調整（2分）**

記事の冒頭と結論は、読者が最も注目する部分です。ここだけ自分の言葉で微調整するだけで、記事全体の印象が大きく変わります。

合計8分。エージェントが書いた記事を人間が8分で仕上げる——これが「自動化」と「品質」を両立させるベストバランスです。

## 5-4. コスト試算：ブログ10記事/月の自動化にいくらかかるか

「理屈はわかった。で、実際にいくらかかるの？」——ここが一番知りたいポイントだろう。ブログ記事を月10本自動生成する場合のコストを、具体的に計算する。

### セッション時間の見積もり方

まず、エージェントの実稼働時間を見積もります。ブログ記事1本の作成にかかるセッション時間は、以下のように分解できます。

| 工程 | 実稼働時間（目安） |
|------|------------------|
| キーワードリサーチ（Web search 3〜5回） | 2〜3分 |
| 構成案の作成 | 1〜2分 |
| 本文執筆（3,000〜5,000字） | 5〜8分 |
| 文体調整エージェント通し | 3〜5分 |
| **合計** | **11〜18分（平均約15分）** |

月10本なら、合計150分（2.5時間）の実稼働時間です。

### 月額コストと想定収益のROI計算

**ランタイム料金:**

2.5時間 × $0.08/時間 = $0.20（約30円）

**API利用料（トークン課金）:**

1記事あたり入力約5,000トークン＋出力約6,000トークンと見積もると、10記事で入力50,000トークン＋出力60,000トークン。Claude Sonnet 4の場合、入力$3/100万トークン＋出力$15/100万トークンなので、約$1.05（約160円）。

**Web search料金:**

1記事あたり3〜5回の検索 × 10記事 = 30〜50回。$10/1,000回なので、$0.30〜0.50（約45〜75円）。

**合計月額コスト:**

| 項目 | 金額 |
|------|------|
| ランタイム料金 | 約30円 |
| API利用料 | 約160円 |
| Web search料金 | 約45〜75円 |
| **合計** | **約235〜265円** |

月額約250円前後で、ブログ記事10本が自動生成されるわけです。ランタイム料金はごくわずかで、コストの大部分はAPI利用料とWeb search料金です。

**想定収益とのROI:**

ブログアフィリエイトの場合、ジャンルや記事の質にもよりますが、1記事あたり月1,000〜5,000円の収益が見込めます（アクセスが安定した場合）。仮に1記事あたり月2,000円とすると、月10本で月20,000円。コスト約250円に対して収益20,000円なら、**ROI（投資対効果）は80倍**です。

もちろん、ブログのアクセスが安定するまでには3〜6か月かかります。しかし記事が積み上がるほどアクセスは増え、ストック資産としての価値は高まります。100記事が積み上がり1記事あたり月1,000円の収益なら、月10万円。運用コスト月250円に対して、手動で100記事書く時間（200時間以上）を考えれば破格です。



### ハンズオン6：ブログ記事作成エージェントをつくる（テンプレート付き）

**ゴール**: キーワードを1つ渡すだけで、SEOを意識したブログ記事を自動生成するエージェントを構築する

**準備**: Anthropic Consoleにログイン済み、APIキーが設定済み、antがインストール済み

**手順：**

1. まず、エージェントを作成します。Consoleの「Agents」→「Create Agent」を開き、以下を設定してください。

   - **Name**: `blog-writer`
   - **Model**: `claude-sonnet-4-6`
   - **Tools**: `Web search`, `Web fetch`, `Write` を有効化
   - **System Prompt**: 以下をコピペ

   ```
   あなたはSEOに精通したプロのWebライターです。
   与えられたキーワードに対して、検索上位を狙える高品質なブログ記事を作成してください。

   ## 作業手順
   1. メインキーワードでWeb検索し、上位5記事の見出し構成を分析する
   2. 関連キーワードを3〜5個洗い出す
   3. 競合分析を踏まえた構成案（h2/h3見出し一覧）を作成する
   4. 構成案に沿って本文を執筆する
   5. 完成した記事をMarkdownファイルとして保存する

   ## 制約条件
   - 文字数: 本文3,000〜5,000字
   - 文体: です・ます調、読者に語りかけるフレンドリーなトーン
   - SEO: メインキーワードをタイトル・最初のh2・本文冒頭100字以内に含める
   - SEO: 関連キーワードを本文中に自然に3回以上使用する
   - Web検索は最大5回までに制限する
   - 他サイトの文章をそのままコピーしない
   - 根拠のない数字や統計データを使わない

   ## 出力形式
   - Markdown形式
   - ファイル名: /home/user/{キーワードをハイフン区切りにしたもの}.md
   - 構造: タイトル（h1）→ リード文 → 本文（h2/h3で構成）→ まとめ
   ```

2. Environmentを作成します。「Environments」→「Create Environment」で以下を設定：
   - **Name**: `blog-env`
   - **ネットワーク設定**: 「Internet Access」をON

3. セッションを作成してエージェントを起動します。ターミナルで以下を実行してください（agent_idとenvironment_idは自分のものに置き換え）：

   ```bash
   ant sessions create \
     --agent-id agent_xxxxxxxx \
     --environment-id env_xxxxxxxx \
     --message "キーワード「副業 在宅 おすすめ」でブログ記事を作成してください"
   ```

4. セッションの完了を待ちます。Consoleの「Sessions」画面でリアルタイムの進捗を確認できます。エージェントがWeb検索→構成案作成→本文執筆の順に自動で進行するのが見えるはずです。

5. セッション完了後、生成された記事ファイルの内容を確認します：

   ```bash
   ant sessions get --session-id session_xxxxxxxx
   ```

**確認ポイント**:
- 記事が3,000〜5,000字の範囲に収まっているか
- タイトルと最初のh2にメインキーワードが含まれているか
- 文体が「です・ます調」で統一されているか

> **つまずきポイント**: 「記事が短すぎる（2,000字以下）」場合は、システムプロンプトの文字数指定を「最低3,000字以上、できれば4,000字以上」のように、下限を強調する書き方に変更してみてください。また、Web searchの結果が英語サイトばかりになる場合は、検索クエリに「日本語」や「日本」を追加するようシステムプロンプトに明記してください。



### ハンズオン7：記事の品質チェックエージェントをつくる

このエージェントはブログ記事の品質チェックに最適化していますが、5-2で紹介した電子書籍の校正にも同じ仕組みを応用できます。

**ゴール**: ハンズオン6で生成した記事を読み込み、AIっぽさの検出と改善提案を行うエージェントを構築する

**準備**: ハンズオン6で生成した記事ファイルがある状態

**手順：**

1. 品質チェック用エージェントを作成します。Consoleの「Agents」→「Create Agent」で以下を設定：

   - **Name**: `article-quality-checker`
   - **Model**: `claude-sonnet-4-6`
   - **Tools**: `Read`, `Write` を有効化
   - **System Prompt**: 以下をコピペ

   ```
   あなたは経験豊富な編集者です。
   AI生成記事の品質をチェックし、「人間が書いたように」改善する提案を行ってください。

   ## チェック項目

   ### 1. AIっぽい表現の検出
   以下のパターンを検出し、改善案を提示してください:
   - 「〜と言えるでしょう」「〜ではないでしょうか」の多用
   - 「まず」「次に」「最後に」の機械的な連続使用
   - 「〜について解説します」「〜を紹介します」のメタ発言
   - 同じ文末表現の3回以上の連続
   - 抽象的な形容詞（「様々な」「多くの」「非常に」）の多用

   ### 2. SEOチェック
   - タイトルにキーワードが含まれているか
   - h2見出しにキーワードまたは関連語が含まれているか
   - メタディスクリプション相当のリード文があるか

   ### 3. 読みやすさチェック
   - 1段落が5文以上になっていないか
   - 1文が80字を超えていないか
   - 漢字比率が高すぎないか（目安: 30%以下）

   ## 出力形式
   以下の形式で出力してください:
   1. 総合評価（100点満点）
   2. AIっぽさスコア（高いほどAIっぽい、10点満点）
   3. 検出された問題のリスト（箇所・問題・改善案）
   4. 改善後の記事全文（修正済みバージョン）

   改善後の記事は /home/user/{元のファイル名}_improved.md として保存してください。
   ```

2. Environmentは、ハンズオン6で作成した `blog-env` を再利用します。

3. セッションを作成して品質チェックを実行します：

   ```bash
   ant sessions create \
     --agent-id agent_xxxxxxxx \
     --environment-id env_xxxxxxxx \
     --message "以下の記事ファイルの品質チェックを行ってください: /home/user/副業-在宅-おすすめ.md"
   ```

4. チェック結果を確認します。「総合評価」「AIっぽさスコア」「問題リスト」「改善後の記事」の4つが出力されているはずです。

5. 改善前と改善後の記事を見比べてみてください。AIっぽい表現がどれだけ修正されたかが一目でわかるはずです。

**確認ポイント**:
- AIっぽさスコアが、改善後の記事では低下しているか
- 指摘された問題が具体的で、改善案が実用的か
- 改善後の記事が自然な日本語になっているか

> **つまずきポイント**: 「チェック結果が大雑把すぎる」場合は、システムプロンプトのチェック項目をさらに細分化してください。たとえば「AIっぽい表現」のパターンを10個以上リストアップすると、検出精度が上がります。また、エージェントが元の記事を読めない場合は、同じEnvironment内にファイルが存在するか確認してください。

**AIライティングと著作権 — 2026年時点の実務ガイドライン**


エージェントにブログ記事や電子書籍を書かせるとき、必ずと言っていいほど気になるのが「著作権」の問題だ。「AIが書いた文章に著作権はあるの？」「AIが書いた記事を売っても大丈夫？」——2026年4月時点での実務的なポイントを整理する。

**1. AI生成物に著作権は発生するのか？**

日本の著作権法では、AI自体が「著者」になることはできません。著作権が認められるためには、人間が「創作的な寄与」を行っている必要があります。文化庁が公表した「AIと著作権に関する考え方について」によれば、単にプロンプトにアイデアを入力しただけでは「創作的寄与」とは認められません。

しかし、本書で解説しているように、システムプロンプトで詳細な構成を設計し、出力を人間が編集・加筆している場合は、人間の創作的寄与が認められる可能性が高いとされています。つまり、「エージェントに書かせっぱなし」ではなく、「人間が設計・監修して仕上げる」というワークフローが、著作権の観点からも重要なのです。

**2. 他者の著作権を侵害するリスク**

より現実的なリスクは、AI生成物が既存の著作物と酷似してしまうケースです。エージェントがWeb searchで収集した情報をもとに記事を書く場合、元の記事の表現がそのまま出力に反映される可能性はゼロではありません。

対策としては、以下の3点が有効です。

- システムプロンプトに「他サイトの文章をそのまま引用・コピーしない」と明記する
- 品質チェックエージェント（ハンズオン7）で、コピーコンテンツの検出を行う
- 公開前に、コピペチェックツール（CopyContentDetectorなど、無料で利用可能）で確認する

**3. 実務上の推奨ワークフロー**

2025年3月に経済産業省と総務省が共同策定した「AI事業者ガイドライン（第1.1版）」では、AI利用者の透明性や著作権配慮が求められています。副業ライターとしては以下を推奨します。

- エージェントの出力は「下書き」として扱い、必ず人間が編集・加筆する
- 公開する記事にはオリジナルの知見や体験を加える
- コピペチェックツールで類似度を確認する（50%以上ならリライト）
- クライアントワークではAI利用の有無を事前に確認する

なお、文化庁は2025年7月に「AIと著作権に関するチェックリスト＆ガイダンス」を追加公表しており、法整備は急速に進んでいます。最新情報は文化庁の公式サイト（bunka.go.jp）で定期的に確認してください。

:::


## まとめ

- **ブログ記事作成は「キーワードリサーチ→構成→執筆」の3ステップで自動化できる**。Web searchツールで競合を分析し、構成案を立て、本文を一気に書き上げるパイプラインをエージェント1つで構築できる
- **電子書籍は「章ごとに別セッション」で書くのが正解**。1回で全部書かせるのではなく、章単位で分割し、校正エージェントで一貫性を担保するのがポイント
- **AIっぽさを消すには「文体統一エージェント＋人間の8分チェック」が効果的**。定型表現の削除と具体的エピソードの追加をエージェントに任せ、独自の知見・ファクトチェック・冒頭と結論の調整を人間が行う
- **月10記事の自動化コストは約250円前後、ROIは最大80倍**。ランタイム約30円＋API約160円＋Web search約50〜75円で、手動なら月20時間かかる作業がほぼゼロになる
- **著作権リスクは「人間の関与」と「コピペチェック」で管理する**。エージェントの出力は下書きとして扱い、人間の編集・加筆を加えることで、著作権の観点でも品質の観点でも安全なワークフローが実現できる
