前章までの第3部では、ライティング、リサーチ、SNS運用、カスタマー対応と、副業ジャンルごとにエージェントを1つずつ構築してきた。それぞれのエージェントが単体で十分な力を発揮することは、実感していただけたはずだ。

しかし、ここからが本書の真骨頂だ。第4部では、これまで構築してきたエージェントを「組み合わせる」ことで、副業の自動化をさらにスケールアップする。

第5章で作ったブログ記事作成エージェントを思い出してほしい。キーワードリサーチから本文執筆まで、1つのエージェントに全工程を任せていた。それでも十分便利だったが、品質を上げようとするほどシステムプロンプトが肥大化し、エージェントの出力が不安定になっていく——そんな経験はなかっただろうか。

この章では、「1つのエージェントに全部やらせる」発想から「複数のエージェントに分業させる」発想へとシフトする。リサーチ専門、執筆専門、校正専門——それぞれの得意分野に特化したエージェントを組み合わせることで、品質・速度・安定性を一段引き上げる方法を示す。

## 9-1. 単体エージェントの限界と「組み合わせ」の力

### 1つのエージェントに全部やらせない設計思想

第5章のブログ記事作成エージェントは、「Web検索でリサーチ→構成案を作成→本文を執筆→SEOチェック」を1つのセッションで行っていました。これは手軽で便利なアプローチですが、タスクが複雑になるにつれて、いくつかの壁にぶつかります。

**壁1：プロンプトの過積載**
「SEOに詳しいWebライターであり、リサーチャーであり、校正者でもある」——1つのシステムプロンプトに複数の役割を詰め込むと、どの役割も中途半端になります。人間の仕事でも同じですよね。「営業も経理も開発もやって」と言われたら、全部が80点になってしまいます。

**壁2：コンテキストウィンドウの圧迫**
エージェントが1つのセッションで長時間動くほど、会話の履歴（コンテキスト）が膨らみます。リサーチで集めた大量の情報が残ったまま執筆に入ると、重要な指示がコンテキストの奥に埋もれ、品質がブレやすくなります。

**壁3：デバッグの難しさ**
出来上がった記事に問題があったとき、「リサーチが甘かったのか」「執筆で方向性がずれたのか」「校正が機能していなかったのか」を切り分けるのが困難です。すべてが1つのセッション内で起きているため、原因の特定に時間がかかります。

これらの壁を突破する答えが「分業」です。会社組織を思い浮かべてください。小さなスタートアップでは1人が何役もこなしますが、事業が成長するにつれて、営業部・開発部・管理部と機能を分けますよね。同じことをエージェントでも行うのです。

### 分業と連携：リサーチ→執筆→校正の3エージェント体制

最もシンプルで効果的なマルチエージェント構成が、「リサーチャー→ライター→エディター」の3エージェント体制です。

**リサーチャーエージェント**は、Web searchツールを使った情報収集に特化します。キーワードを受け取り、競合記事の分析、関連データの収集、ファクトチェックを行い、その結果を構造化されたMarkdownファイル（`research_notes.md`）として保存します。

**ライターエージェント**は、リサーチャーが作成した`research_notes.md`を読み込み、記事の本文を執筆します。システムプロンプトには文体やSEO要件だけを集中的に指定できるので、プロンプトの過積載が起きません。

**エディターエージェント**は、ライターが書いた記事を校正します。誤字脱字、論理の飛躍、事実と異なる記述、SEO要件の充足度をチェックし、修正済みの原稿を出力します。

この3体制のメリットは明確だ。各エージェントのシステムプロンプトがシンプルになり、出力品質が安定する。問題があったときにどの工程で発生したかが一目瞭然になる。さらに、リサーチャーの出力を使い回して別のライターエージェント（たとえばSNS投稿用）に渡せる——つまり、エージェントの「再利用性」が格段に上がるのだ。

著者が実施した比較検証（20セッション）では、1つのエージェントで全工程を処理した場合と3エージェント体制を比較して、記事のSEOスコア（見出し構成・キーワード密度の適切さ）が平均15〜20%向上しました。セッション時間は合計で若干増えるものの（1エージェントで約12分→3エージェントで合計約15分）、品質の改善幅を考えれば十分にペイします。

## 9-2. マルチエージェント構成の設計パターン

マルチエージェント構成には、大きく3つのパターンがあります。副業の内容や目的に応じて、最適なパターンを選びましょう。

### パイプライン型（直列）：A→B→Cの順次実行

最もわかりやすいパターンです。エージェントAの出力をエージェントBの入力にし、Bの出力をCの入力にする。工場のベルトコンベアのように、タスクが順番に流れていきます。

```
[リサーチャー] → research_notes.md → [ライター] → draft.md → [エディター] → final.md
```

**向いているケース**: ブログ記事作成、レポート作成、電子書籍の章執筆など、工程の順序が決まっているタスク。

**メリット**: 設計がシンプル。データの流れが一方向なのでデバッグしやすい。
**デメリット**: 全工程の合計時間がかかる（並列化できない）。

実装方法は驚くほどシンプルです。3つのエージェントをそれぞれ作成し、シェルスクリプトやPythonスクリプトで順番にセッションを起動します。エージェント間の「受け渡し」は、ファイルを介して行います。リサーチャーが`research_notes.md`を書き出し、ライターがそれを読み込む——ただそれだけです。

### ファンアウト型（並列）：複数エージェントの同時実行

複数のエージェントを同時に起動して、それぞれが独立したタスクを並列処理するパターンです。

```
              → [X投稿エージェント]    → x_posts.md
[企画エージェント]  → [Instagram投稿エージェント] → ig_posts.md
              → [note記事エージェント]  → note_draft.md
```

**向いているケース**: SNS運用（複数プラットフォームへの同時投稿作成）、複数クライアントのレポート同時生成など、独立したタスクを一気に処理したいとき。

**メリット**: 処理時間を大幅に短縮できる。3つのタスクを並列に実行すれば、最も遅いタスクの時間だけで済む。
**デメリット**: エージェント間のデータ共有には工夫が必要。同時実行する分、セッション料金が並列に発生する。

たとえば、SNS運用で3プラットフォーム分の投稿を作る場合、直列なら3セッション分の時間がかかりますが、並列なら1セッション分の時間で完了します。仮に1セッションが5分なら、直列で15分・並列で5分。料金は同じ（3セッション分）ですが、処理時間が3分の1になるのは大きなメリットです。

### オーケストレーター型：管理エージェントが子エージェントを制御

最も高度で柔軟なパターンです。「オーケストレーター（指揮者）」となるメインエージェントが、タスクの状況を判断しながら、子エージェント（サブエージェント）を呼び出して作業を割り振ります。

```
[オーケストレーター]
  ├── 判断：まずリサーチが必要 → [リサーチャー]を起動
  ├── リサーチ結果を確認 → 情報不足 → [リサーチャー]を再起動
  ├── 十分なデータが揃った → [ライター]を起動
  └── 記事を確認 → 品質OK → [エディター]を起動 → 完了
```

**向いているケース**: 条件分岐が多いタスク。「リサーチ結果次第で次の工程を変えたい」「品質が基準に達するまでやり直させたい」といった柔軟な制御が必要な場面。

**メリット**: 状況に応じた判断ができる。品質ループ（基準を満たすまで繰り返し）が組める。
**デメリット**: 設計の難易度が高い。オーケストレーター自体のセッション料金が加算される。

Managed Agentsでは、この「オーケストレーター型」をAPIで実現する仕組みとして、サブエージェント機能が用意されています。メインのセッション内からサブエージェントを起動し、その結果をメインエージェントが受け取って次の判断に使う——という流れです。サブエージェントはメインエージェントと同一セッション内で動くため、別途Environmentを用意する必要がなく、手軽に始められます。

## 9-3. Research Preview機能の活用

Managed AgentsのAPIには、2026年4月時点で「Research Preview」（研究プレビュー）として提供されている高度な機能があります。正式リリース前の段階ですが、申請すれば利用でき、マルチエージェント構成を大幅に強化してくれます。ここでは主要な3つの機能を紹介します。

### outcomes：期待する成果物の定義

outcomes（アウトカム）は、エージェントに「何をもって成功とするか」を事前に定義する機能です。通常のシステムプロンプトでも「SEOスコア80点以上の記事を書いて」と指示できますが、outcomesはそれをAPIレベルで構造化します。

outcomesを設定すると、エージェントは自分の出力を成功基準に照らして自己評価し、基準を満たすまで自動的にリトライします。「記事の文字数が3,000字未満なら書き直す」「ファクトチェックで未確認の情報が2つ以上あれば再調査する」といったルールを明確に定義できるのです。

Anthropicの発表によると、outcomes機能により構造化されたファイル生成タスクの成功率の向上が報告されています。特に複雑なタスクほど効果が大きいとのことです。

### multiagent：複数エージェントの協調動作

multiagent（マルチエージェント）は、複数のClaudeインスタンスを協調させる機能です。前述の「サブエージェント」をAPIレベルで正式にサポートするもので、2つの動作モードがあります。

**サブエージェントモード**: メインエージェントのセッション内でサブエージェントを起動します。サブエージェントは独立したコンテキストで動作し、結果だけをメインエージェントに返します。直列のパイプライン型やオーケストレーター型に適しています。トークン消費が比較的少なく、経済的です。

**エージェントチームモード**: 複数のエージェントが独立したセッションを持ち、共有タスクリストを通じて直接コミュニケーションします。ファンアウト型の並列処理に最適です。ただし、複数セッションが同時に走るため、サブエージェントモードよりコストが高くなります。

副業の自動化では、まずサブエージェントモードから始めることをおすすめします。ほとんどのユースケース（記事作成、レポート生成、SNS運用）は、サブエージェントモードで十分にカバーできます。

### memory：セッション間の記憶保持

memory（メモリ）は、エージェントがセッションをまたいで情報を記憶する機能です。通常、エージェントのセッションが終了すると、そのセッション中の会話履歴は失われます。memoryを有効にすると、重要な情報（ユーザーの好み、過去の作業結果、学習した改善ポイントなど）を次のセッションに引き継げます。

副業での活用例を挙げましょう。ブログ記事作成エージェントにmemory機能を設定すると、「前回の記事で読者からの反応が良かったトーン」「過去に使ったキーワードリスト」「クライアントからのフィードバック内容」を記憶し、次回の記事に自動的に反映してくれます。使えば使うほど、あなた好みの出力に近づいていくわけです。

> **注意**: これら3つの機能（outcomes、multiagent、memory）は、2026年4月時点でResearch Previewの段階です。利用するにはAnthropicへのアクセス申請が必要です。本章のハンズオンでは、Research Preview機能を使わずに実現できるパイプライン型の構成を扱います。Research Preview機能が正式リリースされた際には、よりスマートな実装に切り替えることが可能です。

## 9-4. 実践：ブログ運営を完全自動化する3エージェント構成

ここからは、9-1で紹介した「リサーチャー＋ライター＋エディター」の3エージェント体制を、実際にManaged Agentsで構築します。第5章のブログ記事作成エージェントを「進化」させるイメージです。

### リサーチャー＋ライター＋エディターの構築

3つのエージェントを、それぞれ独立した Agent として作成します。

**エージェント1：リサーチャー**

```
名前: blog-researcher
モデル: claude-sonnet-4-6
ツール: Web search, Web fetch, Write
システムプロンプト:
あなたはSEOリサーチの専門家です。
与えられたキーワードについて、以下の調査を行い、
結果を research_notes.md に保存してください。

1. 検索上位10記事のタイトル・見出し構成を調査
2. 関連キーワード（サジェスト）を10個以上収集
3. 競合記事の平均文字数を算出
4. 読者が求めている情報（検索意図）を3つに整理
5. 差別化できるポイントを2つ以上提案

出力形式: Markdown（見出し付き・箇条書き中心）
```

**エージェント2：ライター**

```
名前: blog-writer
モデル: claude-sonnet-4-6
ツール: Read, Write
システムプロンプト:
あなたはプロのWebライターです。
research_notes.md を読み込み、SEOに最適化された
ブログ記事を draft.md に執筆してください。

制約条件:
- 文字数: 3,000〜5,000字
- 文体: です・ます調、フレンドリーなトーン
- 構成: リード文→本文（h2×3〜5個）→まとめ
- SEO: メインキーワードをタイトル・h2・冒頭100字に含める
- research_notes.md の情報を根拠として活用すること
```

**エージェント3：エディター**

```
名前: blog-editor
モデル: claude-sonnet-4-6
ツール: Read, Write, Web search
システムプロンプト:
あなたはプロの編集者です。
draft.md を読み込み、以下の観点で校正・修正し、
final.md として保存してください。

チェック項目:
1. 誤字脱字・文法エラー
2. 論理の飛躍・矛盾
3. 事実情報の正確性（Web searchで裏取り）
4. SEO要件の充足度
5. 読みやすさ（1段落の長さ、接続詞の適切さ）

修正箇所には <!-- EDITED: 理由 --> コメントを付けること
```

ポイントは、各エージェントに与えるツールを最小限にしていることです。リサーチャーにはWeb searchを、ライターにはReadとWriteだけを、エディターにはその両方を。第4章で学んだ「最小限のツール構成から始める」原則を、ここでも適用しています。

### エージェント間のデータ受け渡し設計

3つのエージェントをつなぐ「糊（のり）」が、ファイルによるデータ受け渡しです。Managed Agentsでは、同じEnvironment内で作成されたファイルに複数のセッションからアクセスできます。これを活用して、エージェント間のコミュニケーションを実現します。

データの流れは次のとおりです。

```
keyword.txt（あなたが用意）
      ↓
[リサーチャー] → research_notes.md
      ↓
[ライター]     → draft.md
      ↓
[エディター]   → final.md（完成記事）
```

各ファイルの役割を明確にしておくことが重要です。`research_notes.md`はリサーチ結果の構造化データ、`draft.md`は執筆済み原稿、`final.md`は校正済みの完成記事。このように「ファイル＝エージェント間の契約書」と考えると、設計がブレにくくなります。

それぞれのファイルに含めるべき項目を事前に決めておくのもコツです。たとえば、`research_notes.md`には「メインキーワード」「検索意図」「競合分析」「推奨見出し構成」の4セクションを必ず含める、とルール化しておけば、ライターエージェントが迷わず情報を取り出せます。



### ハンズオン14：リサーチ→執筆→校正の3エージェントパイプラインを構築する

**ゴール**: 3つのエージェントをパイプライン型でつなぎ、キーワードを1つ渡すだけで校正済みのブログ記事が完成する仕組みをつくる

**準備**:
- Anthropic Consoleにログイン済みであること
- CLIツール「ant」がインストール済みであること（第3章参照）
- APIキーが設定済みであること

1. まず、3つのエージェントをConsoleまたはAPIで作成します。前のセクションで紹介した「リサーチャー」「ライター」「エディター」の設定をそれぞれ入力してください。各エージェントのIDをメモしておきます（例：`agent_researcher_xxxx`、`agent_writer_xxxx`、`agent_editor_xxxx`）。

2. 共通のEnvironmentを1つ作成します。3つのエージェントが同じ作業ディレクトリ（ファイル）にアクセスできるようにするためです。

```bash
ant environments create \
  --name "blog-pipeline-env" \
  --config '{"type": "cloud", "networking": {"type": "unrestricted"}}'
```

作成されたEnvironment IDをメモします。

3. パイプラインを実行するスクリプトを作成します。以下の内容を `run_pipeline.sh` として保存してください。

```bash
#!/bin/bash
# ブログ記事3エージェントパイプライン
KEYWORD="$1"
ENV_ID="your_environment_id"
RESEARCHER_ID="your_researcher_agent_id"
WRITER_ID="your_writer_agent_id"
EDITOR_ID="your_editor_agent_id"

echo "=== ステップ1: リサーチ ==="
ant sessions create \
  --agent-id "$RESEARCHER_ID" \
  --environment-id "$ENV_ID" \
  --message "次のキーワードについてリサーチしてください: $KEYWORD" \
  --wait

echo "=== ステップ2: 執筆 ==="
ant sessions create \
  --agent-id "$WRITER_ID" \
  --environment-id "$ENV_ID" \
  --message "research_notes.md を読み込み、ブログ記事を執筆してください" \
  --wait

echo "=== ステップ3: 校正 ==="
ant sessions create \
  --agent-id "$EDITOR_ID" \
  --environment-id "$ENV_ID" \
  --message "draft.md を読み込み、校正してください" \
  --wait

echo "=== 完了！ final.md に記事が保存されました ==="
```

4. スクリプトを実行します。

```bash
chmod +x run_pipeline.sh
./run_pipeline.sh "副業 在宅 おすすめ 2026"
```

5. 実行が完了したら、`final.md`の内容を確認します。リサーチャーが集めた情報が記事に反映されているか、エディターの校正コメント（`<!-- EDITED: -->`）が入っているかをチェックしましょう。

**確認ポイント**:
- `research_notes.md`、`draft.md`、`final.md`の3ファイルが生成されていること
- `final.md`の文字数が3,000〜5,000字の範囲内であること
- 記事にメインキーワードが適切に含まれていること
- エディターの修正コメントが確認できること

> **つまずきポイント**: 「ライターが`research_notes.md`を見つけられない」というエラーが出た場合は、Environmentが同一であることを確認してください。異なるEnvironmentで実行すると、ファイルが共有されません。また、リサーチャーのセッションが完了する前にライターのセッションが始まってしまう場合は、`--wait`フラグが正しく設定されていることを確認してください。このフラグにより、セッションが完了するまで次のコマンドが実行されません。



### コスト試算：3エージェントパイプラインの運用コスト

気になるのはコストだ。3エージェント体制にすると、単体エージェントよりも高くなるのでは——と心配する方も多いだろう。実際の数字で見てみよう。

**単体エージェント（1エージェントで全工程）**:
- セッション時間：約12分
- セッション料金：0.2時間 × $0.08/時間 = $0.016
- API利用料（トークン）：約$0.15
- **1記事あたり合計：約$0.17（約25円）**

**3エージェントパイプライン**:
- リサーチャー：約5分 → $0.007
- ライター：約6分 → $0.008
- エディター：約4分 → $0.005
- セッション料金合計：$0.02
- API利用料（トークン）：約$0.20（各エージェントのコンテキストが小さいため、効率的）
- **1記事あたり合計：約$0.22（約33円）**

差額は1記事あたり約8円。月10記事で約80円の追加コストです。この80円で、記事の品質が向上し、デバッグが容易になり、エージェントの再利用性が高まる。十分にペイする投資だ。

**「自動化しすぎない」勇気 — 人間が介在すべきポイント**


3つのエージェントを組み合わせてパイプラインが動き始めると、「もっと自動化できるのでは」「人間の確認ステップを省けばもっと速くなるのでは」という誘惑に駆られます。たしかに技術的にはできます。しかし、「自動化しすぎない」ことも、実は重要なスキルです。

副業において、人間が必ず介在すべきポイントが3つあります。

**1. 公開前の最終確認**
エディターエージェントが校正した記事でも、そのまま公開するのは危険です。AIは「事実っぽく見える誤情報」を自信満々に書くことがあります。特に、数字・固有名詞・法律に関わる情報は、人間の目で最終チェックしましょう。ブログの読者は「あなた」を信頼して記事を読んでいるのであって、「あなたのエージェント」を信頼しているわけではありません。

**2. クライアントとのコミュニケーション**
SNS運用代行でも、レポート納品でも、クライアントとの関係構築は人間にしかできません。「今月はこんな施策を試してみました」「御社のブランドイメージに合わせてトーンを調整しました」——こうした一言が、継続受注と単発で終わる案件を分けます。

**3. 方向性の判断**
「このキーワードで攻めるべきか」「このクライアントに提案すべきサービスは何か」——戦略レベルの判断は、市場感覚を持つ人間がすべきです。エージェントはデータを集めて分析するのは得意ですが、「何を目指すか」を決めるのはあなたの仕事です。

著者の経験則では、パイプライン全体の工程のうち、80%をエージェントに任せ、20%を人間が担当するバランスが最も安定します。これを「80/20ルール」と呼んでいます。全自動化を目指すより、人間の介在ポイントを戦略的に残すことで、品質とクライアント満足度の両方を維持できます。「自動化しすぎない勇気」——これは、エージェント活用が上手い人ほど大切にしている原則です。

:::


## まとめ

- **1つのエージェントに全部やらせない**。プロンプトの過積載、コンテキストの圧迫、デバッグの困難さを避けるため、役割ごとにエージェントを分けて「分業」させるのが品質向上の鍵
- **マルチエージェント構成には3つのパターンがある**。パイプライン型（直列）はシンプルで初心者向き、ファンアウト型（並列）は処理速度重視、オーケストレーター型は柔軟な制御が可能。まずはパイプライン型から始めよう
- **Research Preview機能（outcomes・multiagent・memory）を活用すれば、マルチエージェント構成をさらに強化できる**。自己評価による品質向上、公式のマルチエージェント協調、セッション間の記憶保持が可能になる（2026年4月時点ではアクセス申請が必要）
- **ファイルを「エージェント間の契約書」として設計する**。各ファイルに含めるべき項目を事前に決めておくことで、データの受け渡しがスムーズになり、パイプライン全体の安定性が向上する
- **「80/20ルール」で自動化しすぎない**。工程の80%をエージェントに任せ、公開前の最終確認・クライアント対応・戦略判断の20%は人間が担当するバランスが、品質と収益を最大化する

次の第10章は、本書の最終章です。ここまで構築してきたエージェントたちを「運用し続ける」ための実践ノウハウ——定期メンテナンス、トラブルシューティング、コスト監視——を解説し、さらに1ジャンルから複数ジャンルへの横展開やチーム運用といったスケールアップ戦略まで踏み込みます。エージェントは「つくって終わり」ではなく、「育てて伸ばす」もの。最終章で、あなたの副業エージェントを長く安定的に稼働させる方法を身につけましょう。
